# PlaneStack — Sprint 96 Pack
**Sprint:** 96  
**Theme:** Reporting Convergence v1 — shared metrics enforcement, Chronicle-owned compare/export, and Operate rollups (Strong target)  
**Status:** Plan (locked for implementation)  
**Date:** 2026-03-08

## Intent
Sprint 96 is an **enforcement-and-expansion sprint over the S95 reporting substrate**, not a new reporting architecture sprint.

Sprint 95 already established the key primitives needed for enterprise-grade reporting convergence:
- canonical project metrics endpoint: `GET /api/v1/ou/:ouKey/p/:projectKey/metrics`
- `/chronicle/metrics/overview` retained only as a thin alias adapter
- shared metrics ownership file: `.planestack/governance/metrics/ownership.json`
- Chronicle hooks for Operate events
- evidence-grade compare/export with integrity hashing

Sprint 96 should capitalize on that work by:
1. extending canonical metrics family coverage
2. enforcing anti-duplication rules in code, tests, and UI
3. standardizing Chronicle-owned compare/export outputs
4. converging Chronicle / Insights / Center / Operate reporting on the same server-side truth
5. proving the above with independent sprint validation evidence

This sprint remains **local-first, deterministic, and pre-deployment**. No deployments, pipelines, or vendor-specific DevOps integrations are introduced.

All Sprint 96 work-producing runs must honor the current filesystem and governance invariants.

- `project_root` and `bridge_root` are distinct filesystem roots. Evidence and governed outputs are written under `project_root` (for example `poc-calculator/.planestack/**`); ephemeral scratch, caches, temp workspaces, and logs remain under `bridge_root`.
- Work-producing runs must set `SPRINT_CODE=S96` and `PS_WORK_PRODUCING=1`.
- Validation and any work-producing run must start with `make -C ps-dev governance-lock-check`.
- Any run that writes governed outputs without the Sprint 96 env contract or writes scratch into `project_root` is invalid for closeout.

## Why Sprint 96 now
The current planning canon explicitly sets **Sprint 96** as the next target, with the fork being **Chronicle/Insights convergence** versus **Operate remote readiness**. The recommended default is convergence first, because Sprint 95 already delivered Operate v2 plus the shared metrics SOT and Chronicle hooks that convergence depends on.

This sequencing is enterprise-correct because it:
- reuses existing canonical reporting primitives instead of creating more
- keeps reporting reproducible from stored data
- prevents UI surfaces from drifting into parallel aggregation logic
- strengthens the **Report** stage of the PlaneStack power loop without touching deployments

## Governing posture (Sprint 96 locks)
These locks are **non-negotiable** for Sprint 96.

### 1) Reuse the existing ownership registry
Do **not** create a new metrics registry.
- Extend `.planestack/governance/metrics/ownership.json` only.
- If fields grow, bump `schema_version`.
- Ownership remains expressed in a **single canonical file**.

### 2) Do not introduce a second contract layer
The metrics contract is expressed through:
- OpenAPI
- JSON schemas
- `.planestack/governance/metrics/ownership.json`

Do **not** create a parallel “metrics family contract” layer.

### 3) No new overlapping metrics endpoints
Sprint 96 may **only extend** the existing canonical metrics endpoint and its schemas.
- Canonical endpoint remains: `GET /api/v1/ou/:ouKey/p/:projectKey/metrics`
- `/chronicle/metrics/overview` remains alias adapter only
- No new overlapping Chronicle / Insights / Center metrics endpoints are allowed

### 4) Compare/export are Chronicle-owned operations
Compare and export are generated by Chronicle ops:
- `chronicle:compare`
- `chronicle:export`

They must write governed artifacts under:
- `.planestack/governance/evidence/s96/chronicle/compare/YYYYMMDDThhmmssZ_<6-8hex>.json`
- `.planestack/governance/evidence/s96/chronicle/export/YYYYMMDDThhmmssZ_<6-8hex>.json`

UI reads governed results; UI does not compute compare/export outcomes.

Deterministic artifact id format for compare/export outputs:
- `YYYYMMDDThhmmssZ_<6-8hex>`
- compare writes to `.planestack/governance/evidence/s96/chronicle/compare/<compare_id>.json`
- export writes to `.planestack/governance/evidence/s96/chronicle/export/<export_id>.json`

### 5) Operate does not get a separate reporting engine
- **Operate** remains the execution cockpit
- **Chronicle** remains the reporting substrate
- Operate may display shared metrics from the canonical endpoint, but it must not introduce separate aggregation logic

### 6) UI cannot aggregate covered metric families
Covered metric families for Sprint 96 v1 are:
- `operate`
- `chronicle`
- `rc`
- `parity`
- `incidents`

For covered metric families:
- UI aggregation helpers are forbidden
- UI must call the SOT endpoint and render returned results
- Enforcement is required via lint + UI smoke proof
- Lint applies only to families declared in `.planestack/governance/metrics/ownership.json`

### 7) UI scope stays thin
- No new top-level nav
- Enhance existing Chronicle and Insights surfaces only
- Center changes only where overlapping metrics must swap to the shared SOT
- No broad redesign of IA or navigation

### 8) Validation remains two-track
- Add `sprint96-validate`
- Keep `e2e-gate` separate and still required
- `sprint96-validate` must not embed `e2e-gate`

### 9) Reuse the S95 integrity rule exactly
All S96 reporting artifacts reuse the existing integrity rule:
- `sha256` of canonical JSON excluding `generated_at` and `integrity`

Do not introduce a new hashing scheme.

### 10) Reporting outputs must be evidence-grade
For compare/export outputs:
- include `source_refs`
- include integrity fields using the S95 rule
- include provenance such as `generated_at`, metric family/version, filters/window, and dataset hash where applicable

## Scope included
Sprint 96 includes the following epics.

### EPIC 96.1 — Shared metrics SOT enforcement
Objective: extend and enforce the canonical shared metrics endpoint across reporting surfaces.

#### S96-101 Extend canonical metrics family coverage
- Extend `GET /api/v1/ou/:ouKey/p/:projectKey/metrics` to cover overlapping families needed by:
  - Chronicle trust/provenance surfaces
  - Insights / Analytics destination cards
  - overlapping Center executive snapshot cards
  - Operate trend cards that are reporting-oriented
- Keep response schemas versioned and deterministic.

Acceptance:
- Required overlapping metric families are available from the canonical endpoint.
- No new overlapping endpoint is introduced.

#### S96-102 Extend ownership registry (do not replace it)
- Extend `.planestack/governance/metrics/ownership.json` with any new fields needed for Sprint 96 such as:
  - metric family id
  - owning subsystem
  - consuming surfaces
  - source evidence roots / source refs
  - no-data semantics
  - schema version
- Preserve single-file ownership governance.

Acceptance:
- A single ownership file remains authoritative.
- New/updated metric families are present and schema-versioned.

#### S96-103 Ownership lint and duplicate-path enforcement
- Add/extend a lint/check that fails when:
  - a covered metric family is implemented in more than one aggregation path
  - UI aggregation helpers are used for covered metric families
  - a new overlapping endpoint is introduced
- Add an endpoint-overlap guard in `sprint96-validate` that fails if any new endpoint overlaps an existing covered metrics scope.
- Allowed behavior is limited to extending `GET /api/v1/ou/:ouKey/p/:projectKey/metrics`; `/chronicle/metrics/overview` remains alias adapter only.
- Wire this into `sprint96-validate`.

Acceptance:
- Duplicate aggregation paths fail validation.
- Covered UI aggregation helpers fail validation.

## Acceptance criteria (Sprint 96)
1. Covered overlapping metric families are served by the canonical metrics endpoint only.
2. `.planestack/governance/metrics/ownership.json` remains the single ownership file and is extended rather than replaced.
3. `/chronicle/metrics/overview` remains a thin alias adapter only.
4. No covered UI surface performs local aggregation for covered metric families.
5. Chronicle and Insights show the same overlapping numbers for the same window/filter inputs because they are served from the same SOT endpoint.
6. Operate trend reporting is derived from Chronicle rollups / canonical metrics families rather than a separate Operate aggregation engine.
7. `chronicle:compare` writes governed compare artifacts with `insufficient_data` handling.
8. `chronicle:export` writes evidence-grade export artifacts with integrity + source refs.
9. `sprint96-validate` passes independently of `e2e-gate`.
10. `e2e-gate` also passes and remains a separate required signal.

## Risks and mitigations
### Risk 1 — Endpoint sprawl by good intentions
Mitigation:
- hard lock on canonical metrics endpoint reuse
- ownership lint fails overlapping implementations

### Risk 2 — UI silently recomputes “just one card”
Mitigation:
- forbid covered aggregation helpers in UI
- require lint + UI/network smoke evidence

### Risk 3 — Compare/export become feature fluff
Mitigation:
- Chronicle-owned ops only
- governed artifacts only
- provenance + integrity required

### Risk 4 — Operate reporting forks into its own engine
Mitigation:
- explicit lock: Operate is cockpit, Chronicle is reporting substrate
- shared endpoint use only

### Risk 5 — Validation becomes ambiguous
Mitigation:
- keep `sprint96-validate` and `e2e-gate` independent
- list required evidence artifacts up front
